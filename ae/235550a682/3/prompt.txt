Dossier onboarding: route rename, post-signin direction, inline settings, multi-attachment references, prompt wiring

Context

Today, sign-in leaves the writer with no next step (/signin/ just shows a "signed in" header with nothing to click), the create-vs-refine route naming is asymmetric (/onboarding/ vs /refining/), settings the writer actually cares about at dossier-creation time (interview style, AI provider, citation defaults) live only on a separate /settings/ page, the dossier can only absorb a single pasted/uploaded starter document, and none of the agent prompts (persona room, rubric, synthesis) ever see anything beyond the 7 brief-interview answers — bibliography/reference material is invisible to the AI. This plan fixes all four in one pass since they touch the same files (the onboarding route and ProjectBrief).

4. Multi-document/link attachments with a one-line "why"

No existing data model fits (BibEntry is per-folio Lix storage, mismatched with the global, pre-folio ProjectBrief). Add attachments directly to the brief.

- src/types/index.ts: add
export interface DossierAttachment {
  id: string;
  kind: "document" | "link";
  title: string;
  url?: string;      // kind === "link" (or optional source URL for a document)
  text?: string;      // kind === "document", capped ~2000 chars
  why: string;        // required one-liner
  addedAt: number;
}
- Extend ProjectBrief with attachments: DossierAttachment[] (default [] everywhere it's constructed).
- src/utils/anti-tabula-rasa.ts: update createProjectBrief(answers, existing) to accept/merge an attachments param; loadProjectBrief/saveProjectBrief need no signature change (they round-trip the whole object).
- New component src/components/onboarding/dossier-attachments-editor.tsx: add/list/remove attachments. Each add-row: kind toggle (Document / Link), title field, conditionally a textarea + the existing hidden-<input type="file"> pattern (copy the idiom from anti-tabula-rasa.tsx's current import step — hidden input + label, onChange$ reads file.text(), no FileReader) for "Document", or a URL field for "Link", plus the required one-line "why" field. List shows added items with a remove (×) button. Cap document text client-side at ~2000 chars (truncate with a note, same spirit as the prompt-side cap in §5).
- Wire into both interview components:
  - AntiTabulaRasa (src/components/onboarding/anti-tabula-rasa.tsx): add a new step to STEPS (new kind: "attachments", alongside the existing kind: "import" step) rendering <DossierAttachmentsEditor>; thread attachments into the existing onSubmit$ signature ((answers, existingMaterial, filename, attachments)).
  - ConversationalInterview (src/components/onboarding/conversational-interview.tsx): render <DossierAttachmentsEditor> inline on the synthesis-review screen (next to the "Take it to the desk" button), pass the collected attachments into onComplete$ alongside the synthesized ProjectInterviewAnswers — easiest is to widen onComplete$'s payload to { answers, attachments } or add a second store-level field the parent route reads after acceptSynthesis.
  - src/routes/dossier/create/index.tsx completeOnboarding$ and src/routes/dossier/refine/index.tsx's submit handler both pass attachments into createProjectBrief/saveProjectBrief so refine mode can also edit existing attachments (initialBrief/initialAnswers already flow into both components — attachments should flow the same way via a new initialAttachments prop sourced from store.brief.attachments).

5. Wire the dossier (incl. attachments) into agent prompts

ProjectBrief already flows everywhere (AgentRequest.brief: ProjectBrief | null in convex/agentPrompts.ts:42-59, populated at every call site in personas-panel.tsx, rubric-panel.tsx, and convex/agents.ts). Since attachments now live on ProjectBrief, no AgentRequest shape change, no new Convex action args, no client call-site changes needed — only the prompt builders need to serialize the new field.

- convex/agentPrompts.ts: add a attachmentsBlock(brief) helper and splice it into:
  - buildUserPrompt (line 141) — append after the existing briefBlock, e.g. REFERENCE MATERIAL (writer-supplied, use to ground feedback — do not treat as the draft itself), one bullet per attachment: - [link|doc] "{title}" — {why} plus (for documents) a fenced excerpt capped at ~500 chars per item, ~2000 chars total across all attachments (separate, tighter cap than the editor's storage cap — keep prompt token cost bounded even if a future caller stores more).
  - generateLocalFeedback (line 220) — same data available locally, can skip elaborate serialization (it's the deterministic fallback) but should at least count/name attachments if present, matching the existing brief-fallback tone.
  - buildSynthesisPrompt (line 286) and buildRubricReviewPrompt (line 307) — same attachmentsBlock helper, since both already take brief: ProjectBrief | null.
- Convex schema/validator check: grep convex/agents.ts for the v.object(...) validator(s) that shape the brief arg on the action endpoints (the three const req: AgentRequest = {...} sites at lines 586, 826, 880 imply matching args: v.object({ brief: v.object({...}) }) declarations) and add the attachments field there too — Convex strictly validates args, so this is required even though no TS call site changes are needed otherwise.
- No changes needed in personas-panel.tsx/rubric-panel.tsx beyond what already exists (brief: brief ?? null already carries the new field once ProjectBrief itself is widened upstream).
