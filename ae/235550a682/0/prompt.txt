Context

When the writer replies to an editor's inline persona note (the popover opened by
clicking a .twyne-persona-note span in the manuscript), the persona's AI answer is
generated correctly but only ever appears in the Cast side-panel thread — never back
in the text. The inline popover closes on send, so from the writer's point of view "in
the text" nothing happens. The user wants the model to respond in the text: the note
popover should become a live conversation (writer reply → typing indicator → persona
response → continue the back-and-forth) without leaving the editor.

The backend already supports this — submitReply(noteId, askPersona) in
personas-panel.tsx files the reply and calls client AI (runClientAgent("persona-reply", …),
src/utils/ai-client.ts:250) or Convex api.agents.runPersona (convex/agents.ts:597,
instruction:"elaborate"), both of which accept priorMessages + userMessage for
multi-turn. The only gap is surfacing the thread back into the editor popover. The
editor and panel are separate components that already communicate over a twyne:* window
CustomEvent bus, so we extend that bus rather than add shared state.

Approach

Keep submitReply as the single source of truth for filing replies + generating
responses. Add events so the panel streams thread state to the editor, and make the
editor popover render the thread + typing indicator and stay open.

typing indicator around the AI call.
- Panel → editor twyne:persona-reply-error { noteId, message } — surfaces the existing
store.conveneError cases inline (today they fail silently from the popover's view).
- Editor → panel twyne:request-persona-thread { noteId } — fired when a popover opens so
prior history renders immediately.
